1. Technology Selection and Platform Migration 
The original Specification planned to use Unity Engine with C# as the primary development 
platform. During early-stage development, it became evident that native Android development 
(Java + XML) offered significant advantages for this project. 
The final technology stack is: 
 Platform: Android (Compile SDK 36) 
 Language: Java 11 + XML 
 IDE: Android Studio 
 Database: SQLite 
 Key Libraries: HanLP (1.8.6), pinyin4j (2.5.0), Gson (2.8.9), iFlyTek SparkChain SDK

6. Pronunciation Quiz with iFlyTek ISE 
The Pronunciation Quiz has been implemented with professional-grade speech evaluation powered 
by iFlyTek's Intelligent Speech Evaluation (ISE) engine: 
a) Core Gameplay: Display a target word/sentence with Pinyin; user holds the microphone button 
to record their pronunciation; receive instant multi-dimensional feedback 
b) Speech Evaluation Engine: IseEvaluator.class manages the complete ISE WebSocket lifecycle: 
 WebSocket connection establishment with iFlyTek API authentication 
 Audio stream chunking and transmission 
 XML result parsing for multi-dimensional scores 
c) Score Mapping: Raw ISE score (0–100) is normalized to a game score (0–10) for consistent 
user experience 
d) UI Features: 
 Hold-to-record microphone button 
 Real-time VoiceWaveView volume visualization during recording 
 Detailed ISE feedback display after evaluation 
 Retry mechanism for re-attempting pronunciation

存在问题 Problems:   
1) HanLP Library Size on Mobile Devices 
Running a full NLP library on mobile devices presented memory and performance challenges. 
The standard HanLP distribution includes large dictionary files that could impact app size and 
loading time. 
 
2) Speech Recognition Accuracy for Non-native Speakers 
The iFlyTek ISE engine is primarily trained on native Chinese speakers. Non-native learners 
with varying accents may receive lower scores, which could affect user motivation and the 
fairness of the scoring system. 
 
3) Unreasonable Difficulty Classification of Questions Across Three Game Modes 
The current difficulty grading for questions in the Character-Pinyin Matching, Pronunciation 
Quiz and upcoming Word Puzzle games is based on subjective manual labeling, lacking a 
unified and data-driven difficulty evaluation standard. This leads to inconsistent difficulty 
calibration between different game modes—for example, some Medium-level questions in the 
Pronunciation Quiz are actually equivalent to Hard-level in the Character-Pinyin Matching 
game. Meanwhile, individual game modes have internal difficulty confusion, such as overly 
simple Hard-level questions or overly challenging Easy-level questions in partial game 
sections, which seriously affects the user's learning and gaming experience and fails to achieve 
the intended progressive difficulty guidance for learners.

拟采取的办法 Solutions: 
1) HanLP Optimization 
Adopted the portable version (1.8.6) of HanLP, which is optimized for smaller footprints. Pre-
processed segmentation results during data loading via SeedDataLoader and cached results in 
the SQLite database, so HanLP does not need to run during gameplay. 
 
2) Pronunciation Scoring Normalization 
Mapped raw ISE scores (0–100) to a normalized game score (0–10) with learner-friendly 
thresholds. Provided multi-dimensional feedback (accuracy, fluency, completeness, tone) so 
users can understand specific areas for improvement rather than just seeing a single number. 
 
3) Standardized Difficulty Classification System for Cross-Game Questions 
First, establish a unified multi-dimensional difficulty evaluation index system applicable to all 
three game modes, including core indicators such as Chinese character stroke count, word 
frequency of vocabulary, pinyin/tone complexity, sentence length and syntactic difficulty, and 
assign scientific weight coefficients to each indicator according to the characteristics of 
different gameplays. Second, use a data-driven labeling method to replace manual subjective 
labeling: based on the established index system, write an automated scoring script to calculate 
the difficulty score of each question, and divide the questions into Easy, Medium and Hard 
levels with clear score thresholds to ensure cross-game difficulty consistency.
